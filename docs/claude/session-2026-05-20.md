## 1. One mat with num_runs, respect user direction (`794382f`)

Two bugs at once:

- `autoFillMatSerpentine` ignored the user's direction choice from the
  workflow panel — it always called `_suggestDirection(room)` (PCA).
  User's "horisontalt"/"vertikalt" selection is stored on
  `S.varmefolie.direction` (the picker is shared with varmefolie),
  but the mat code never read it. Now reads userDir first, PCA as
  fallback.
- Variable-length mats produced N independent mat objects (one per
  run) — physically wrong, since a roll-style mat is ONE product that
  gets cut into a serpentine. Both fixed and variable length now
  produce a single mat object with `num_runs` set.

–65 net lines.

## 2. Variant picker — initial policy (`d65d69f`)

Added a picker that walks the product family (matched on
`product_family`, `mat_width_mm`, and a name-prefix fallback) and
picked the **largest variant whose total length ≤ room's ideal need**.
Idea: an undersized mat with leftover area is fine; an oversized mat
is wasteful.

This was the wrong direction — see step 5.

## 3. Cap to physical mat length + flexible family match (`30b3c49`)

User reported "1.6 cm gap" still wrong because the mat label said
"0.5×2m" while the layout claimed 5 × 336 cm = 16.8 m of mat — physically
impossible from a 200 cm product. Two fixes:

- `runLengthCm` is now capped to `totalLengthCm` for fixed-length mats.
  Without this, a 0.5×2m product could end up rendered as 5 runs of
  336 cm each.
- Family matching is more tolerant: when `product_family` is missing
  (common in Supabase), strip the size suffix from the name
  ("EcoMat 60T 0.5×4m" → "EcoMat 60T") and group siblings that way.

## 4. Infer total length from area+width (`c8170ca`)

User's catalog had `mat_area_m2: 1` and `mat_width_mm: 500` but no
`mat_total_length_mm`, so the code treated the product as
variable-length and skipped the physical-length cap. Result: 8 × 240 cm
of mat from a 2 m product.

New helper `_matTotalLengthMm(prod)`:

- returns `prod.mat_total_length_mm` if set
- otherwise derives from `mat_area_m2 * 1_000_000 / mat_width_mm`
- returns null if neither is available

All readers of the field route through the helper:
`_upcRenderMatResults`, `autoFillMatSerpentine` variant picker,
`_matChangeRuns` (max-runs guard), and the ctxbar mat label.

## 5. Flip variant policy + even-gap distribution (`84c9ea3`)

User: "i eksemplet vedlagt ville jeg heller lagt 7 stk bredder ved
siden av hverandre … 1,6 cm mellomrom. mattene kan legges helt intil
hverandre."

Two changes:

- **Flipped variant policy**: now picks the **smallest variant whose
  length ≥ ideal need**, so all room runs get covered (with some
  excess). Falls back to largest variant if no size covers. The
  previous policy left a 240×420 room filled by a 12 m mat with only
  5 of 7 possible runs.
- **Even gap distribution**: leftover across-axis space is split as
  equal gaps between mats instead of clumped to the side margins.
  `mat.gap_cm` is now stored on the mat object so the renderer can
  honor it; both `_matTotalWidthCm` and `drawMats` prefer
  `mat.gap_cm` over the product's `min_gap_mm` fallback.

Verified:

  240×420 room → 4 runs × 400 cm in 16 m mat, 10 cm gap
  377×246 room → 7 runs × 224 cm in 16 m mat, 2.83 cm gap

(User's "1.6 cm" calculation assumed a 370 cm room; my output is on
377 — math checks out otherwise.)

## 6. 'Målsett Matte' dim chain (`36bee09`)

New per-room flag `room.matDimSnap` (mirrors `stripDimSnap` /
`cableDimSnap`). New `toggleMatDimSnap()` and `_toggleMatDimSnap(roomId)`
helpers. New `_drawMatDimChainForRoom(room)` rendering two chains:

- **Across-chain** (perpendicular to runs, on bottom for 'v' /
  right for 'h'): `wall | gap | mat | gap | mat | ... | wall` —
  shows the even-distributed gaps and the wall margins explicitly.
- **Along-chain** (parallel to runs, on left for 'v' / top for 'h'):
  `wall | start-margin | length | end-margin | wall` — confirms the
  mat doesn't run to the walls.

Mat runs are reconstructed in world coords from `mat.x_cm`, `mat.y_cm`,
`mat.width_cm`, `mat.length_cm`, `mat.num_runs`, `mat.gap_cm`, and
`mat.rotation_deg`. Hooked into `drawAllDimAnnotations`. Room-dim
baseOff now adds +65 cm when mat-dim is on (same offset used for
cable-dim) so the chains don't collide.

`Målsett Matte` chip added to the mat ctxbar, matching styling of
the existing strip/cable chips.

## Known fine-tuning items deferred

User mentioned "noen fin tuninger vi må gjøre senere":

- The "smallest variant ≥ need" policy can pick a noticeably larger
  variant than the user might want (e.g. 20 m mat for 16.32 m need).
  Worth a UI affordance: surface excess in the auto-fill banner so
  the user can opt down to the next size.
- Direction picker still lives on `S.varmefolie.direction` even
  when the product is a mat. Works in practice (shared widget), but
  semantically a separate `S.varmematte.direction` would be cleaner.
- Mat across-chain currently always orients to bottom/right. For
  rooms where mats sit closer to the top/left walls, flipping the
  chain side per-room would read better.
- Mat dim chain's `_drawExtLine` still uses dashed extension lines.
  We removed them for room-dim earlier but kept them on the cable
  and strip chains. Confirm with the user whether to keep or remove.
