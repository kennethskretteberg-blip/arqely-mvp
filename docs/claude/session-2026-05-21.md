## 1. Måslett Rom-button collision (`9fed29a`)

Two paths to "Målsett Rom" did different things:

- Sidebar 📐 → `_toggleRoomDimSnap()` → `room.dimSnap` (drives the wall-by-wall
  dim chain via `_drawDimAnnotationsForRoom`).
- Ctxbar "Målsett Rom" → `autoDimRoom()` → appended dashed user-dim lines to
  `room.dims`.

Clicking both stacked the two systems and looked wrong. Both now call
`_toggleRoomDimSnap(roomId)` and read `room.dimSnap`. `autoDimRoom()` is left
in place but no longer called from any UI path. Added "📋 Rom-label" chip to
the room-selected ctxbar, mirroring the sidebar 📋 (`toggleRoomCard`).

Also fixed `selectMultiCables`: it returned `'1x'` the moment a CC-valid
single cable existed, even if it only covered 67% of requested power
(InFloor 17T 3400W in a 39 m² / 5070 W room). Now only commits to `'1x'`
when single coverage ≥ 90% of `desiredW`; otherwise falls through to
multi-cable evaluation so the UI can offer both single picks and a 2×
option.

## 2. PDF export overhaul (`8b65e07`, `3edca3f`, `ebb68c6`, `35a51c0`, `9a37009`)

Iterative work. The starting point was that PDFs were missing what users
actually needed.

`8b65e07`: Per-room PDF page now renders the **main canvas** with a
temporary view fitted to the room (capture-to-dataURL via
`canvas.toDataURL`) rather than the simple offscreen renderer. Pixel-perfect
match: cable U-turns, labels, selection-aware visuals, dim chains — all
inherited from `render()`. State is fully restored after capture. Cover
table now shows actual product names + CVA/EL numbers + count instead of
generic "Kabel", and `Romtegner — arqely.com` branding is gone.

`3edca3f`: Bug from the previous commit — `panX`/`panY` are world-cm coords
in `w2s()` (the world point that maps to canvas center), not pixel offsets.
I had set them in pixel space, which panned the room far off-screen and
captured only the background grid. Pan now uses the room centroid in world
coords. Also temporarily set `S.ui.showGrid = false` so the PDF shows
everything except the grid, restored afterward.

`ebb68c6`: User feedback that the room-name label, area badge, corner dots,
transform gizmo and info cards were noise on the PDF (already in the
project-overview table). Added `S.ui._pdfMode` flag, set by
`_renderRoomToImage` before render and restored after. `drawRooms` gates
four spots on it (Pass-1 label, Pass-2 label, corner dots, transform gizmo
call). `drawRoomCards` returns early when set.

`35a51c0`: Material list shows piece count AND meters for every linear
product (Folie, Kabel, Matte) — not just kabel as before. Plate stays
"N stk" since not linear. Mat aggregation also fixed: `num_runs > 1` is one
physical product cut into serpentine, so total cable meters and covered
area now scale by `num_runs` (old code undercounted by 4–8× on auto-filled
rooms).

`9a37009`: Two unrelated fixes that piled up.

The big one — drawRooms Pass-2 had `const cx` / `const cy` moved inside the
`if (!_pdfMode)` block during the gating refactor. But the wall-label loop
further down (V1/V2/V3/V4 placement on the inside) also reads `cx`/`cy` to
pick the inside direction. With them out of scope, `render()` silently
bailed mid-function once a room was selected — strips, hindringer and dim
chains all disappeared on click. Moved the centroid to the outer scope; the
label block only gates the actual text draw now.

Also added floating dim lines so the user can drop control measurements
right after setting the background scale (before any room is drawn). New
`S.floatingDims` array; dim-tool commit falls back to it when no room is
selected; `drawUserDimLines` concatenates them with `room.dims`; new
Mållinje chip appears in the empty-canvas ctxbar when the bg is loaded and
calibrated.

## 4. Hindring polygon — pick which half (`f5973fc`)

When a user draws a hindring polyline that starts and ends on room walls,
the polygon effectively splits the room and either half could plausibly be
the obstacle. The previous `_traceWallPath` always picked the shorter
corner-trace, which guessed wrong in plenty of layouts.

`_traceWallPath` grew a `which` param (`'fwd' | 'bwd' | 'shortest'`,
default keeps old behaviour). The hindring-polygon wall-snap close path
builds BOTH candidate polygons and stashes them on `S.ui.hindringChoice`
instead of committing. `drawPreview` renders an overlay: candidate A
(cyan dashed) and B (orange dashed), each labelled with a small "klikk
for å velge" hint. Mousedown in `hindring-polygon` mode, while
`hindringChoice` is set, runs `ptInPoly` against each half and commits
the polygon containing the click. Ambiguous clicks (both/neither) do
nothing — wait for an unambiguous one. `exitDrawing` clears the state.

## 5. Polygon length labels (`558c27b`)

The per-segment "X cm" label drawn while building a room or hindring
polygon was 11px no-background — easy to lose against the bg image or
other overlays. Bumped to 16px bold and wrapped each label in a rounded
pill (white in light theme, near-black in dark) with a draw-color stroke.
Label keeps its baseline-rotation along the segment.

## 6. Strip placement + dim-line delete (`cdba2c0`)

Two unrelated fixes folded into one commit because they piled up at the
same time.

**Strip placement bug.** `doSemiAutoStep()` pushed strips with the full
room along-length, ignoring hindringer and forbidden zones — resulting
strip overlapped obstacles. Auto-fill at `autoAddStrips()` already routed
pushes through `computeClippedSegments()` correctly; semi-auto was the
odd one out.

Added shared helper `_pushStripsAtPos(roomId, productId, direction,
pos_cm)` right above `doSemiAutoStep`. It runs `computeClippedSegments`,
pushes one strip per returned segment, invalidates the strip cache,
returns the created strips. Callers around it own cursor advance / snap /
overlap. Both `doSemiAutoStep` while-loops (low→high, high→low) now call
the helper. `autoAddStrips`'s forward + backward branches also routed
through it for consistency.

Verified against a 4×6 m room with two horizontal hindringer in the
middle: placing a vertical strip at x=80 produces 3 strips (top,
between, bottom), all sharing `pos_cm=80` with different
`start_cm/length_cm`. Each can be selected and edited independently.

Drag-from-palette was deliberately left alone — its post-push magnetic-
snap logic is tangled with the single-strip-result shape, larger
refactor needed.

**Dim-line delete.** Click selection only worked for `room.dims`;
`floatingDims` were unreachable. Added them to the click hit-test in
canvas mousedown. New `deleteDim(dimId)` helper walks both collections,
removes the id from wherever it lives, pops the just-pushed undo entry
if nothing matched, clears the selection if pointed at the deleted dim,
re-renders. Delete key now calls the helper. New `_hitDimLine` and
`dimCtxMenu` helpers wire a right-click menu (`✕ Slett mållinje`) into
the canvas `contextmenu` listener between hindring and wall checks.
