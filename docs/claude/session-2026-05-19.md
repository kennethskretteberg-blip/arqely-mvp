## 3. Cables follow the room on move/rotate (`ad43234`)

User reported that rotating a room left cables stuck at their old
positions. The room-transform handler at `_thDragging` mouseup was
moving/rotating `room.points`, hindrings, and `S.strips` — but cables
were completely ignored.

Cable runs are stored as `{ pos_cm, segments[] }` aligned to the
direction axis. That representation can't faithfully represent a
non-axis-aligned serpentine pattern, so for arbitrary rotation angles
matrix-rotating the run coords would corrupt the geometry. Instead,
after the drag ends we call `autoFillCable(roomId, productId)` for each
cable in the moved/rotated room. Cable `id`, `roomId`, and `productId`
are preserved so undo/redo and selection don't break; the rest of the
cable object is replaced.

This means cables snap to new positions on mouseup rather than animating
through the drag — acceptable trade-off given the structural limitation.
